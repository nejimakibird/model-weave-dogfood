# Model Weave V0.8 Documentation Policy

English Version: [English](../V0.8-documentation-policy.md)

## 目的

この文書は、Model Weave の V0.8 ドキュメントポリシーを定義します。

V0.8 では、Custom / Mermaid レンダリングや `render_mode` に関する現在のレンダリング方針を整理します。
V0.8 では、Model Weave のドキュメント、特に FORMAT 仕様書をどのような構成で整理・維持するかを定義します。

目的は、Model Weave のドキュメントを次の人たちにとって使いやすくすることです。

* 初めて使うユーザー
* メンテナー
* Model Weave Markdown を生成するAIアシスタント
* 生成されたモデルをレビューする人
* ソースコードや仕様書から設計資料を再構築する人

このポリシーは、ドキュメント構成に関するものです。
プラグインの動作、パーサーの動作、レンダラーの動作、既存モデルファイルの互換性を変更するものではありません。

## ドキュメントポリシーのバージョンとプラグインバージョン

ドキュメントポリシーのバージョンと、プラグインのバージョンは別の概念です。

例:

* Plugin version: `0.1.x`
* Rendering policy: `V0.8`
* Documentation / FORMAT policy: `V0.8`

この文書でいう `V0.8` は、ドキュメントポリシーのバージョンです。
ここに書かれている内容が、すべてプラグインバージョン `0.8` の機能である、という意味ではありません。

## ドキュメントの役割

Model Weave のドキュメントは、役割ごとに分けて整理します。

### README

ルートの `README-ja.md` は、最初に読む入口です。

README-ja では次の内容を説明します。

* Model Weave とは何か
* 何に使うのか
* どう使い始めるのか
* 最初に試すコマンド
* 詳細ドキュメントへの導線

README に、すべてのコマンドやすべての FORMAT に関する詳細は記載されていません。

### Getting Started

Getting Started は、最初に使うためのチュートリアルです。

使用開始できる最小限の手順を説明しています。

* Model Weave をインストールして有効化する
* サンプルモデルを開く
* Obsidian のコマンドパレットを開く
* プレビューコマンドを実行する
* 診断を確認する
* PNG出力を試す
* Source Links がある場合は確認する

### Command Guide

Command Guide では、Model Weave が提供する Obsidian コマンドを説明しています。

コマンドは目的別に分類されています。

* preview / index / export 系コマンド
* テンプレート挿入コマンド
* 編集補助コマンド
* 設定に関する補足

コマンド名は、 Obsidian のコマンドパレットで表示される内容に合わせてあります。
`Model Weave:` 接頭辞付きで記述してあります。

### FORMAT仕様書

FORMAT仕様書は、Model Weave Markdown モデルファイルの構造を定義しています。

FORMAT仕様書では次の内容を説明します。

* そのフォーマットが何に使われるか
* 必須・任意の frontmatter
* 対応するセクション
* 期待されるテーブルヘッダー
* 安全な記述ルール
* よくあるミス
* AI生成時の注意
* 関連サンプル

FORMAT仕様書では、テーブルヘッダーや対応セクションを詳しく説明しています。

### Samples

Samples は、具体的な例を示しています。

ユーザーが FORMAT 仕様を実際の Markdown ファイルでどう使うかを理解するために役立つと思います。

Samples は FORMAT 仕様書の代替ではありません。
FORMAT 仕様書が構造を定義し、Samples は実用例を示します。

### Dogfoodモデル

Dogfood モデルは、高度な実例であり、実装を理解するための地図でもあります。

Dogfood モデルには次のような内容が含まれる場合があります。

* Source Links
* 実装メモ
* 将来構想
* Model Weave 自身の内部構造
* AI生成またはAI支援で作られたモデル

Dogfood モデルは実践的な例として有用ですが、実験中または将来構想の内容を含むことがあります。
そのため、Dogfood モデルの記述内容は現在の仕様と一致しない部分がある点に注意が必要です。

## FORMATドキュメントの標準構成

各 FORMAT ドキュメントは、原則として次の構成にします。

### What this is for

そのフォーマットの目的を説明します。

このセクションでは次の疑問に答えます。

* このフォーマットは何を表すのか
* どのようなときに使うのか
* このフォーマットで表すべきではないものは何か

### Minimal example

最小限の有用な例を示します。

初めて使う人がコピーして理解できる程度の短い例にします。

必須でない限り、高度なオプションは含めません。

### Full example

必要に応じて、より実用的な例を示します。

Full example には次のような内容を含めても構いません。

* 複数行の例
* 関連
* notes
* Source Links
* render mode の例
* 関連モデル参照

ただし、Full example を大きくしすぎないでください。
大きな例は `samples/` に置きます。

### Frontmatter

対応する frontmatter 項目を説明します。

含める内容は次の通りです。

* 必須項目
* 任意項目
* 期待される値
* Wikilink の安全な記述
* 例

パーサーが対応していない frontmatter 項目は、その旨を明示します。

### Sections

対応する Markdown セクションを説明します。

各セクションについて、次の内容を説明します。

* 何を表すのか
* 必須か任意か
* どのように解析されるか
* プレビューや診断にどう影響するか

### Tables

期待されるテーブルヘッダーを正確に記述します。

これは、AI支援による生成で特に重要です。

各テーブルについて、次の内容を含めます。

* セクション名
* 期待されるヘッダー
* 各列の意味
* 必須列 / 任意列
* 安全な値
* 参照や Wikilink に関する注意

未対応の列を記述してはいけません。

### Common mistakes

ユーザーやAIアシスタントがよく起こすミスを記載します。

例:

* 未対応の列を追加する
* 別FORMATのテーブルヘッダーを混ぜる
* モデル概念を混同する
* Markdownテーブルとして危険な記法を使う
* 実装済みセクションに将来構想を書く
* frontmatter内のWikilinkをクォートしない

### AI generation notes

AIアシスタントがこのFORMATを安全に生成するための注意を書きます。

このセクションには、次のようなルールを含めます。

* テーブルヘッダーを正確に保つ
* 未対応の列を追加しない
* 型名は単純なスカラー型を優先する
* 補足情報は `notes` に書く
* 追加情報は任意セクションに分ける
* 実装済みの動作と将来構想を区別する
* 必要に応じて Source Links を使い、追跡可能性を確保する

### Related samples

`samples/` 配下の関連ファイルまたはディレクトリにリンクします。

関連サンプルは、ユーザーが動作する例をすぐに見つけられるようにするためのものです。

## AIに安全なMarkdown記述ルール

AIで Model Weave Markdown を生成または更新する場合は、次のルールに従います。

### テーブルヘッダーを正確に保つ

AIアシスタントは、FORMATドキュメントで明示されていない限り、テーブル列を追加、削除、並べ替え、改名してはいけません。

人間には自然に見えるテーブルでも、パーサーにとっては不正な場合があります。

### 未定義の列を追加しない

対象FORMATが明示的に対応していない限り、`ref`, `rule`, `description`, `notes`, `source`, `target` などの列を追加しないでください。

補足情報が有用な場合は、次の場所に書きます。

* 既存の `notes` 列
* 対応している任意セクション
* `## Notes`
* `## Source Links`
* 別のモデルファイル
* 必要に応じて TODO やガイドファイル

### frontmatter内のWikilinkはクォートする

frontmatter は通常の Markdown 本文ではなく YAML です。

frontmatter 内で Wikilink を使う場合は、クォートします。

推奨:

```yaml
source: "[[DATA-EXAMPLE]]"
```

避ける例:

```yaml
source: [[DATA-EXAMPLE]]
```

リンク機能が不要な場合は、プレーンなIDの方が安全な場合もあります。

```yaml
source: DATA-EXAMPLE
```

### テーブル内でWikilinkエイリアスを避ける

次のようなテーブルセル値は避けます。

```md
[[DATA-USER|User]]
```

代わりに、通常のWikilinkを使います。

```md
[[DATA-USER]]
```

Wikilinkエイリアスには `|` が含まれるため、Markdownテーブル解析に影響する可能性があります。

### テーブルセル内で生のパイプ文字を避ける

次のような値は避けます。

```text
string | null
A | B | C
```

より安全な表記を使います。

```text
string
A / B / C
```

任意性は、FORMATが対応している場合は `required`, `not_null`, `notes` などで表現します。

### 型名は単純なスカラー型を優先する

テーブルセル内では、次のような単純な型名を優先します。

* `string`
* `number`
* `boolean`
* `date`
* `datetime`

次のような複雑な型表現は避けます。

* `string[]`
* `Array<string>`
* `Optional<string>`
* `string | null`

複数性や任意性が重要な場合は、対応している列または `notes` に記述します。

### 実装済みの動作と将来構想を分ける

Dogfood モデルや設計メモには、将来構想が含まれる場合があります。

FORMATドキュメントを生成または更新するときは、次を明確に区別します。

* 実装済みの動作
* ドキュメント化された動作
* 実験中の動作
* 将来構想
* notes や TODO

将来構想を、実装済みの動作として記述してはいけません。

### Source Linksで追跡可能性を確保する

モデルを実装ファイル、テスト、設定、SQL、インターフェース仕様、サンプルデータなどへ結びつける必要がある場合は、`## Source Links` を使います。

Source Links は、AIが生成した設計モデルを元のソースで検証するための手がかりになります。

## 安定性ラベル

FORMATドキュメントでは、そのFORMATがどの位置づけなのかを示します。

* stable / primary
* experimental / evolving
* internal / dogfood-only
* deprecated / legacy

Stable フォーマットは、多くのユーザーにとっての推奨開始点です。

Experimental フォーマットは利用可能な場合がありますが、ドキュメントやモデリング上の慣習が変わる可能性があります。

## V0.8レンダリングポリシーとの関係

V0.8 rendering policy は、現在のレンダラー動作を説明します。

この V0.8 documentation policy は、ドキュメント構造と記述ルールを扱います。

使い分けは次の通りです。

* [V0.8 rendering policy](../V0.8-rendering-policy.md): レンダラー動作、`render_mode`、Custom / Mermaid 方針、Mermaid Detail の利用可否、Mermaid Source / Debug の動作。
* この文書: ドキュメント構成、FORMAT仕様書の構成、AIに安全なMarkdown記述ルール。

## メンテナンスルール

FORMATドキュメントを更新するときは、次を確認します。

1. 現在の実装またはパーサー動作を確認する。
2. 既存サンプルを確認する。
3. 対応しているテーブルヘッダーを正確に保つ。
4. 未対応の列を記述しない。
5. 理解しやすい小さな例を追加する。
6. 大きな例は埋め込まず、サンプルへリンクする。
7. AIで生成されやすいFORMATには、AI generation notes を追加する。
8. stable と experimental の区分を明確に保つ。

## レビュー用チェックリスト

ドキュメント変更を受け入れる前に、次を確認します。

* そのFORMATが何に使われるか説明しているか
* 最小例があるか
* frontmatter項目が説明されているか
* 対応セクションが説明されているか
* テーブルヘッダーが正確か
* 未対応列を記述していないか
* よくあるミスが説明されているか
* AI生成時の注意があるか
* 関連サンプルへリンクしているか
* 実装済みの動作と将来構想が区別されているか
* リンクが有効か

## まとめ

V0.8 documentation policy は、Model Weave の FORMAT ドキュメントを予測しやすく、安全に使えるものにするための方針です。

目的は、現在のFORMATを説明するだけではありません。
AIアシスタントが正しく生成しやすく、人間がレビューしやすいドキュメント体系にすることです。
