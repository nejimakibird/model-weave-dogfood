# V0.7 Rendering Policy

## Purpose

This document defines the V0.7 rendering direction for Model Weave.

V0.7 focuses on clarifying the rendering architecture before further implementation work. In particular, it defines:

- DFD Mermaid formalization
- `render_mode` introduction
- Mermaid/custom renderer responsibility split
- DFD simplified local object direction
- future removal of the legacy DFD custom renderer

This document is architectural guidance. It does not change runtime behavior by itself.

## Current Context

Model Weave currently has multiple preview and rendering paths.

- ER / Class / Screen Preview already use custom rendering for detailed review views.
- DFD is planned to move to Mermaid `flowchart LR` as the formal renderer.
- The Model Weave Viewer already has shared UI behavior such as zoom, fit, `100%`, pan, PNG export, diagnostics, and upper/lower resizable layout.
- Some views need detailed interaction and row-level review, while other views need large overview rendering.

Because of this split, V0.7 needs a documented rendering policy before further format expansion.

## Source of Truth Policy

- Markdown files are always the source of truth.
- Mermaid code is generated output.
- SVG / PNG / preview views are derived outputs.
- Users should not edit Mermaid output as the design source.
- The plugin should parse Model Weave Markdown into an internal model before rendering.

Mermaid is a rendering path, not a design authoring format.

## Renderer Responsibility Split

### Mermaid renderer

Mermaid is primarily responsible for overview rendering.

- Overview diagrams
- Large graph readability
- DFD
- Future screen transition diagram
- Future app_process simple flow
- Potential `class_diagram` / `er_diagram` overview mode

### Custom renderer

Custom rendering is primarily responsible for detailed review views.

- Detailed review views
- Single object views
- Rich table views
- Screen Preview
- ER entity details
- Class details
- Click navigation / row jump heavy views
- Diagnostics-rich views

### Shared rule

- Mermaid and custom renderer are both derived renderers.
- Choice of renderer should not change the Markdown source format.

## render_mode Policy

Supported values:

- `auto`
- `custom`
- `mermaid`

Behavior:

- Missing `render_mode` is treated as `auto`.
- Unknown `render_mode` should produce a Warning and fall back to `auto`.
- `auto` means “use the default renderer for this format”. It is not itself a renderer.

Priority order:

1. Toolbar renderer selection  
   Temporary UI override. Does not write back to the file.
2. `frontmatter.render_mode`  
   File-level renderer preference.
3. `settings.defaultRenderMode`  
   Plugin-level default.
4. Format default  
   Renderer default by format type.

Toolbar visibility policy:

- Show the renderer selector only when multiple meaningful modes exist for the format.
- Keep the selector hidden for Mermaid-first DFD views.
- Keep the selector hidden for table/text-only views.

Initial format defaults:

- `class`: `custom`
- `class_diagram`: `custom` for now; may support Mermaid later
- `er_entity`: `custom`
- `er_diagram`: `custom` for now; may support Mermaid later
- `dfd_object`: definition/detail object view
- `dfd_diagram`: `mermaid`
- `screen`: `custom`
- `app_process`: table/text view for now
- `codeset`: table/text view
- `message`: table/text view
- `mapping`: table/text view
- `rule`: table/text view

## DFD Rendering Policy

- The formal renderer for `dfd_diagram` is Mermaid `flowchart LR`.
- The legacy DFD custom renderer is planned for removal after Mermaid DFD is confirmed stable.
- The DFD parser and internal model remain.
- `dfd_object` remains as a definition/detail object, not as a switchable Mermaid/custom diagram view.
- Shared Viewer components remain.
- The DFD Mermaid renderer should generate stable internal node IDs.
- Display labels should remain separate from internal Mermaid node IDs.

Node label priority:

1. `Objects.label`
2. resolved ref name
3. `Objects.id`
4. raw ref

Edge label policy:

- Edge labels should use `Flows.data`.

Failure behavior:

- Mermaid generation failure must not crash the Viewer.
- It should report diagnostics and show a safe fallback message.

## DFD Simplified Objects Policy

V0.7 also defines the future direction for simplified `Objects` rows in `dfd_diagram`.

Planned columns:

- `id`
- `label`
- `kind`
- `ref`
- `notes`

Meaning:

- `id`: local object ID inside the diagram
- `label`: display label
- `kind`: `external` / `process` / `datastore`
- `ref`: optional link to `dfd_object`
- `notes`: optional notes

Rules:

- `ref` is optional.
- If `ref` is empty, treat the row as a local diagram object.
- A `ref`-empty local object is allowed and should produce Note at most.
- If `ref` is present but unresolved, produce Warning or Error.
- `Objects.id` must be unique within the diagram.
- `Flows.from` / `Flows.to` should resolve to `Objects.id` or `Objects.ref`.
- `Flows.from` / `Flows.to` should generally refer to objects listed in `Objects`.
- Avoid silently creating missing objects from `Flows`.

Compatibility note:

- The old `Objects` table format with only `ref` / `notes` remains accepted for compatibility.
- The new `id / label / kind / ref / notes` format is the preferred V0.7 authoring form.

## Diagnostics Policy

- `ref` missing by design is not an error for DFD local objects.
- `ref` present but unresolved is a diagnostic issue.
- Unknown `render_mode` is Warning.
- Mermaid generation failure is Error or Warning, but should not crash the Viewer.
- Unsupported renderer for a format should fall back safely.
- Diagnostics should distinguish `Error` / `Warning` / `Note`.

## Mermaid Security and Interaction Policy

- Mermaid click callback syntax should not be the primary interaction mechanism.
- Prefer generated Mermaid + SVG post-process for navigation if needed.
- Obsidian file navigation and row jump should remain Model Weave responsibility.
- Mermaid should be used as a rendering/layout engine, not as a behavior source.
- Avoid requiring unsafe Mermaid settings only for navigation.
- Generated Mermaid should come from Model Weave internal models, not arbitrary user-provided executable behavior.

## Viewer / PNG Export Policy

- Viewer toolbar renderer selection is temporary.
- `settings.defaultRenderMode` is the plugin-level default and does not rewrite Markdown.
- `defaultZoom`, `fontSize`, and `nodeDensity` are viewer settings, not source-model settings.
- Zoom / fit / `100%` / pan should be shared where possible.
- Upper/lower resizable layout should remain shared Viewer behavior.
- PNG Export should export diagram body only.
- Toolbar, diagnostics panel, lower information area, and resize handle should not be included in PNG.
- For Mermaid diagrams, exported PNG should include the Mermaid-rendered diagram body.
- Export should fit the full diagram, not only the current zoom/pan state.

## Core / Adapter Direction

- V0.7 should start reducing Obsidian API dependency in core logic.
- Parser / resolver / validator / graph model builder / renderer input should move toward plain types.
- Obsidian-specific `TFile` / `App` / `Vault` / `WorkspaceLeaf` should be isolated in adapter or service layers where feasible.
- This V0.7 docs task does not perform large directory restructuring.
- VS Code extension is not part of V0.7 implementation.
- This is an architectural direction, not a full refactor in this task.

## Non-goals for V0.7-01

- implement `render_mode`
- implement DFD local objects
- implement Mermaid renderer changes
- delete the legacy DFD renderer
- add settings UI
- refactor directories
